# 程序分析实验一：活变量分析和迭代求解器

## 一、实验目标

1. 理解并掌握活变量分析（Live Variable Analysis）的原理和数据流方程。
2. 使用 Java 在 Tai-e 静态分析框架中实现活变量分析的具体逻辑。
3. 实现通用的向后迭代数据流求解器（Iterative Solver）。

## 二、实验理论

### 2.1 活变量分析的定义

活变量分析是一种经典的**向后数据流分析**（Backward Dataflow Analysis）。一个变量 `v` 在程序点 `p` 是**活的（live）**，当且仅当：从 `p` 点出发到程序出口的某条路径上，`v` 在被重新定值之前被使用。简言之，如果变量在将来还会被用到，它就是活的。

### 2.2 数据流方程

对于每个基本块（语句）`B`，活变量分析定义如下：

- **OUT[B]** = 所有后继节点 `S` 的 IN[S] 的并集：
  $$OUT[B] = \bigcup_{S \in succ(B)} IN[S]$$

- **IN[B]** 的转移函数：
  $$IN[B] = use_B \cup (OUT[B] - def_B)$$

  - `use_B`：语句 B 中在被定值之前就使用的变量集合
  - `def_B`：语句 B 中被定值（赋值）的变量

- **边界条件**：对于 CFG 的出口节点，OUT = ∅（空集）
- **初始条件**：所有非边界节点的 IN 和 OUT 均初始化为 ∅

- **交汇操作（meet）**：对于多条路径的汇合，取并集（Union）

## 三、实验实现

### 3.1 LiveVariableAnalysis.java

```java
package pascal.taie.analysis.dataflow.analysis;

import pascal.taie.analysis.dataflow.fact.SetFact;
import pascal.taie.analysis.graph.cfg.CFG;
import pascal.taie.config.AnalysisConfig;
import pascal.taie.ir.exp.Var;
import pascal.taie.ir.stmt.Stmt;

public class LiveVariableAnalysis extends
        AbstractDataflowAnalysis<Stmt, SetFact<Var>> {

    public static final String ID = "livevar";

    public LiveVariableAnalysis(AnalysisConfig config) {
        super(config);
    }

    // 活变量分析是向后分析
    @Override
    public boolean isForward() {
        return false;
    }

    // 边界条件：出口节点的 IN 为空集
    @Override
    public SetFact<Var> newBoundaryFact(CFG<Stmt> cfg) {
        return new SetFact<>();
    }

    // 初始条件：非边界节点的初始 fact 为空集
    @Override
    public SetFact<Var> newInitialFact() {
        return new SetFact<>();
    }

    // 交汇操作：将 fact 并入 target（并集）
    // 直接在 target 上进行 union，避免创建新对象
    @Override
    public void meetInto(SetFact<Var> fact, SetFact<Var> target) {
        target.union(fact);
    }

    // 转移函数：IN[B] = use_B ∪ (OUT[B] - def_B)
    // 注意：在向后分析中，参数 in 对应 OUT[B]，参数 out 对应 IN[B]
    @Override
    public boolean transferNode(Stmt stmt, SetFact<Var> in, SetFact<Var> out) {
        // 保存旧的 out（即旧的 IN[B]）用于判断是否发生变化
        SetFact<Var> oldOut = out.copy();

        // 先将 out 设为 in（即 OUT[B]），然后从中去除 def，最后加上 use
        out.set(in); // out = OUT[B]

        // 移除被定值的变量（kill）：out = OUT[B] - def_B
        if (stmt.getDef().isPresent() && stmt.getDef().get() instanceof Var) {
            out.remove((Var) stmt.getDef().get());
        }

        // 添加使用的变量（generate）：out = use_B ∪ (OUT[B] - def_B)
        for (var exp : stmt.getUses()) {
            if (exp instanceof Var) {
                out.add((Var) exp);
            }
        }

        // 返回 IN[B] 是否发生了变化
        return !out.equals(oldOut);
    }
}
```

**关键实现说明：**

1. **`isForward()`** 返回 `false`，表示活变量分析是向后分析，数据流从出口向入口方向传播。

2. **`newBoundaryFact()`** 和 **`newInitialFact()`** 都返回空的 `SetFact`。在向后分析中，边界节点是 CFG 的出口节点，出口之后没有任何变量是活的，因此为空集。非边界节点初始化为空集（底元素 ⊥），因为活变量分析中格的顶元素是全集，底元素是空集。

3. **`meetInto()`** 实现并集操作：`target.union(fact)`。直接在 `target` 上原地操作，无需创建新对象，效率更高。

4. **`transferNode()`** 实现转移函数 `IN[B] = use_B ∪ (OUT[B] - def_B)`。注意参数映射：
   - `in` 参数 = OUT[B]（来自后继节点的数据流信息）
   - `out` 参数 = IN[B]（传递给前驱节点的数据流信息）

   实现步骤：先将 `out` 设为 `in`（OUT[B]），然后移除 `def` 中的变量，最后添加 `use` 中的变量。需要特别注意 `getDef()` 和 `getUses()` 返回的是 `LValue/RValue`，只有当它们是 `Var` 类型时才进行操作。

### 3.2 IterativeSolver.java（及 Solver.initializeBackward）

#### Solver.java — `initializeBackward` 方法

```java
protected void initializeBackward(CFG<Node> cfg, DataflowResult<Node, Fact> result) {
    // 设置边界条件：出口节点的 IN fact 为 newBoundaryFact（空集）
    result.setInFact(cfg.getExit(), analysis.newBoundaryFact(cfg));
    // 所有其他节点（非出口节点）的 IN 和 OUT 都初始化为 newInitialFact（空集）
    for (Node node : cfg) {
        if (!cfg.isExit(node)) {
            result.setOutFact(node, analysis.newInitialFact());
            result.setInFact(node, analysis.newInitialFact());
        }
    }
}
```

**说明：** 在向后分析中，边界节点是 CFG 的出口节点。出口节点的 IN fact（对应数据流方程中的 OUT[exit]）设为 `newBoundaryFact()`，即空集。其余所有节点的 IN 和 OUT 初始化为 `newInitialFact()`，也是空集。这样迭代求解器可以从空集开始，逐步收敛到不动点。

#### IterativeSolver.java — `doSolveBackward` 方法

```java
@Override
protected void doSolveBackward(CFG<Node> cfg, DataflowResult<Node, Fact> result) {
    boolean changed;
    do {
        changed = false;
        for (Node node : cfg) {
            // 跳过出口节点（其 IN 已经是边界条件）
            if (cfg.isExit(node)) {
                continue;
            }
            // 第一步：计算 OUT[B] = ∪ IN[S]，遍历所有后继节点
            Fact out = result.getOutFact(node);
            for (Node succ : cfg.getSuccsOf(node)) {
                analysis.meetInto(result.getInFact(succ), out);
            }
            // 第二步：应用转移函数 IN[B] = use_B ∪ (OUT[B] - def_B)
            // 如果 IN[B] 发生变化，标记 changed = true
            if (analysis.transferNode(node, result.getOutFact(node), result.getInFact(node))) {
                changed = true;
            }
        }
    } while (changed); // 直到没有节点的 IN 集合发生变化
}
```

**关键实现说明：**

1. **迭代循环**：使用 `do-while` 循环，在每次迭代中遍历 CFG 的所有节点（跳过出口节点）。

2. **计算 OUT[B]**：对于每个节点，遍历其所有后继节点，使用 `meetInto` 将后继节点的 IN fact 并入当前节点的 OUT fact。由于 `meetInto` 直接在 `out` 上操作（原地并集），所以不需要每次重新计算所有后继的并集，只需将新的信息并入即可——这就是 `meetInto` 的优化所在。

3. **应用转移函数**：调用 `transferNode` 计算 IN[B]。如果返回 `true`（表示 IN[B] 发生了变化），设置 `changed = true`，继续下一轮迭代。

4. **终止条件**：当一轮迭代中没有任何节点的 IN 集合发生变化时，算法达到不动点，终止迭代。

## 五、实验总结

通过本次实验，我深入理解了向后数据流分析的实现原理。在活变量分析中，数据流信息从程序出口向入口方向传播，转移函数的核心公式为 `IN[B] = use_B ∪ (OUT[B] - def_B)`。实现时需要特别注意参数的映射关系：在向后分析中，传入 `transferNode` 的 `in` 参数实际对应数据流方程中的 OUT[B]，而 `out` 参数对应 IN[B]。

在迭代求解器的实现中，`meetInto` 的设计体现了增量优化的思想——通过直接在 `target` 上执行并集操作，避免了每次迭代都重新计算所有后继节点 IN 集合的完整并集。当只有部分后继节点发生变化时，只需将变化的部分并入目标集合即可，这显著提升了求解效率。同时，使用 `changed` 标志位检测不动点收敛，确保算法在数据流信息稳定后正确终止。

此外，在实现过程中我也注意到了类型安全的问题：`stmt.getDef()` 返回的 `LValue` 和 `stmt.getUses()` 返回的 `RValue` 并不一定是 `Var` 类型（如数组访问、字段访问等），需要通过 `instanceof` 检查进行过滤，避免 `ClassCastException`。这一细节提醒了我在处理 IR 层次结构时需要严谨对待类型转换。
